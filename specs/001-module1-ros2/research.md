# Research: Module 1 — The Robotic Nervous System (ROS 2)

**Branch**: `001-module1-ros2` | **Date**: 2026-03-07
**Phase**: 0 — Outline & Research

All unknowns from Technical Context resolved below.

## Decision 1: Docusaurus Version

**Decision**: Docusaurus 3.x (latest stable, currently 3.7.x)

**Rationale**:
- Docusaurus 3.x is the current maintained branch with React 18 and MDX 3 support.
- Docusaurus 2.x reached maintenance-only status; no new features.
- MDX 3 enables richer component embedding for future chatbot integration.
- Native Mermaid plugin shipped in core since v2.4, stable in v3.

**Alternatives considered**:
- Docusaurus 2.x: Rejected — maintenance-only, MDX 2 only, React 17
- MkDocs (Material): Rejected — Python ecosystem, no native MDX/React component support
  needed for future chatbot embedding
- GitBook: Rejected — vendor lock-in, no self-hosted GitHub Pages control

---

## Decision 2: Docusaurus Installation Location

**Decision**: `book/` subdirectory at repository root

**Rationale**:
- Repository root already contains SDD artifacts (`.specify/`, `specs/`, `history/`);
  mixing Docusaurus files (node_modules, package.json) at root creates noise.
- `book/` isolates the frontend application cleanly; CI/CD paths are explicit.
- `cd book && npm run build` is unambiguous in GitHub Actions.

**Alternatives considered**:
- Root installation: Rejected — pollutes repo root with `node_modules`, `package.json`;
  confusing for contributors working on backend/chatbot in future
- `website/` subdir: Rejected — `book/` is more semantically meaningful for this project

---

## Decision 3: Sidebar Organization

**Decision**: Autogenerated sidebar from directory structure using `_category_.json`
and `sidebar_position` frontmatter

**Rationale**:
- Docusaurus autogenerated sidebars eliminate manual `sidebars.js` maintenance.
- `_category_.json` controls module label, position, and collapsibility per directory.
- `sidebar_position: N` in each chapter's frontmatter controls chapter ordering.
- Adding new chapters requires only a new MDX file — no config edits needed.

**Alternatives considered**:
- Manual `sidebars.js` entries: Rejected — requires updating config for every new chapter;
  error-prone for a multi-module book
- Single flat docs directory: Rejected — doesn't support multi-module hierarchy

---

## Decision 4: Diagram Technology

**Decision**: Mermaid diagrams via `@docusaurus/theme-mermaid` (built into Docusaurus 3)

**Rationale**:
- Mermaid is natively supported via `themeConfig.mermaid` in Docusaurus 3 — zero extra
  install beyond enabling the theme.
- Mermaid diagrams are version-controlled as text (no binary image files).
- Supports flowcharts (architecture), sequence diagrams (message flow), and graph
  diagrams (URDF hierarchy) — all needed for Module 1.

**Alternatives considered**:
- Static PNG/SVG images: Rejected as primary — not version-control-friendly; hard to edit
- draw.io XML: Rejected — requires external tool; not renderable in browser without plugin
- Excalidraw: Rejected — outputs raster images; not text-based

---

## Decision 5: GitHub Pages Deployment

**Decision**: GitHub Actions workflow using `actions/deploy-pages` with `docusaurus build`
output from the `book/` directory

**Rationale**:
- GitHub Pages from `gh-pages` branch via GitHub Actions is the standard pattern for
  Docusaurus projects.
- `docusaurus.config.js` sets `url`, `baseUrl`, `organizationName`, `projectName` for
  correct GitHub Pages URLs.
- No additional hosting cost — aligns with constitution free-tier constraint.

**Workflow**:
```yaml
# .github/workflows/deploy.yml (high-level)
on: push to main
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - checkout
      - setup-node 18
      - cd book && npm ci && npm run build
      - deploy build/ to gh-pages branch
```

**Alternatives considered**:
- Netlify: Rejected — adds vendor dependency; GitHub Pages is free and sufficient
- Manual `npm run deploy`: Rejected — not automated; requires local SSH key setup

## Decision 6: ROS 2 Target Distribution

**Decision**: ROS 2 Humble Hawksbill (LTS, supported until May 2027)

**Rationale**:
- Most widely deployed ROS 2 LTS at time of authoring.
- Python 3.10 rclpy API is stable and well-documented.
- All code examples verified against the official Humble API reference.
- EOL May 2027 gives adequate shelf life for the book.

**Alternatives considered**:
- ROS 2 Iron Irwini: Shorter support window (EOL Nov 2024) — rejected
- ROS 2 Jazzy Jalisco (LTS): Newer (May 2024+), less widely deployed — acceptable
  future upgrade path but Humble more universally accessible now

---

## Decision 7: Code Example Validation Strategy

**Decision**: Python syntax validated via `python -m py_compile`; URDF validated via
`xmllint --noout`; both run as pre-merge checks documented in quickstart.md

**Rationale**:
- `py_compile` catches syntax errors without requiring a ROS 2 runtime environment.
- `xmllint` is available in all standard Linux environments; validates XML well-formedness.
- Full runtime testing of rclpy code requires a ROS 2 installation — documented as
  optional in quickstart.md; syntax check is the enforceable gate.
