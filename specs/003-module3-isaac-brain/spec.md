# Feature Specification: Module 3 — The AI-Robot Brain (NVIDIA Isaac)

**Feature Branch**: `003-module3-isaac-brain`
**Created**: 2026-03-09
**Status**: Draft
**Input**: Create a Docusaurus module covering NVIDIA Isaac tools — Isaac Sim for photorealistic simulation and synthetic data generation (Chapter 1), Isaac ROS perception with GPU-accelerated VSLAM (Chapter 2), and Nav2 autonomous navigation with path planning and obstacle avoidance (Chapter 3).

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Understand Isaac Sim and Synthetic Data Generation (Priority: P1)

A robotics student reads Chapter 1 and understands how NVIDIA Isaac Sim provides a photorealistic simulation environment for training AI perception models using synthetic data — and why synthetic data generation is the foundation of the Isaac AI training pipeline.

**Why this priority**: Isaac Sim is the starting point of the Isaac ecosystem. Students must understand synthetic data generation before they can understand why Isaac ROS perception (Chapter 2) achieves such high accuracy — it was trained on Isaac Sim data.

**Independent Test**: Chapter 1 can be read alone. A student should answer: "What is synthetic data and why is it used?", "What does the Replicator tool do?", "How does Isaac Sim differ from Gazebo?"

**Acceptance Scenarios**:

1. **Given** a student reads Chapter 1, **When** asked what synthetic data is, **Then** they explain that it is photorealistic, annotated training data generated by a simulator rather than collected from the real world — and that it eliminates the cost and risk of real-world data collection.
2. **Given** a student reads the Replicator section, **When** asked what domain randomization does, **Then** they explain that it varies lighting, textures, object poses, and camera angles to make AI models robust to real-world variation.
3. **Given** a student reads the Isaac Sim vs Gazebo comparison, **When** asked when to use each, **Then** they say Isaac Sim for photorealistic AI training data; Gazebo for physics-accurate functional robot testing.

---

### User Story 2 — Understand Isaac ROS GPU-Accelerated Perception (Priority: P2)

A robotics student reads Chapter 2 and understands how Isaac ROS GEMs (GPU-Enabled Modules) deliver hardware-accelerated robotics perception pipelines — specifically Visual SLAM (cuVSLAM) for real-time pose estimation and the NITROS zero-copy transport layer that eliminates CPU bottlenecks.

**Why this priority**: P2 because perception is the bridge between the simulation training pipeline (Chapter 1) and the navigation system (Chapter 3). A robot cannot navigate without knowing where it is — that is cuVSLAM's job.

**Independent Test**: Chapter 2 can be read independently. A student should explain: "What is a GEM?", "What does cuVSLAM do?", "Why does NITROS improve pipeline performance?"

**Acceptance Scenarios**:

1. **Given** a student reads Chapter 2, **When** asked what a GEM is, **Then** they explain it is a GPU-Enabled Module — a pre-built, hardware-accelerated ROS 2 package from NVIDIA that processes sensor data on the GPU instead of the CPU.
2. **Given** a student reads the cuVSLAM section, **When** asked to describe the VSLAM pipeline, **Then** they name the five steps: feature extraction, feature matching, visual odometry, map update, and loop closure.
3. **Given** a student reads the NITROS section, **When** asked why NITROS improves performance, **Then** they explain it eliminates CPU-GPU memory copies by passing tensor data directly between GPU-resident nodes via a shared memory transport.

---

### User Story 3 — Understand Nav2 Autonomous Navigation (Priority: P3)

A robotics student reads Chapter 3 and understands how the Nav2 navigation stack enables autonomous robot movement — from costmap-based obstacle representation through global path planning with NavFn to local trajectory execution with DWB, including recovery behaviors.

**Why this priority**: P3 because Nav2 is the action system that consumes the robot pose from cuVSLAM (Chapter 2) and executes the physical navigation commands that the VLA planner (Module 4) will request.

**Independent Test**: Chapter 3 can be read independently with Module 1 (ROS 2) knowledge. A student should explain: "What is a costmap?", "What is the difference between the global and local planner?", "What happens when Nav2 encounters an obstacle?"

**Acceptance Scenarios**:

1. **Given** a student reads Chapter 3, **When** asked what a costmap is, **Then** they explain it is a 2D grid map where each cell has a cost value representing how safe it is for the robot to occupy that cell — free space has low cost, obstacles have high cost.
2. **Given** a student reads the planner section, **When** asked the difference between global and local planner, **Then** they explain: global planner (NavFn/Smac) computes a complete path from start to goal on the costmap; local planner (DWB Controller) executes the path in real time, reacting to dynamic obstacles.
3. **Given** a student completes Chapter 3, **When** asked what happens when the robot detects an unexpected obstacle, **Then** they describe: local planner replans around it; if stuck, Nav2 triggers a recovery behavior (spin, back up, clear costmap).

---

### Edge Cases

- A student arrives at Chapter 2 without reading Chapter 1: prerequisite admonition must clarify that Isaac ROS perception models are trained on Isaac Sim synthetic data — the two tools are explicitly connected.
- A student arrives at Chapter 3 without Module 1 (ROS 2): prerequisite admonition must link to Module 1 and explain that Nav2 publishes to `/cmd_vel` and subscribes to `/odom`, `/scan`, and `/map`.
- Isaac Sim hardware requirements: students may not have an NVIDIA GPU. Chapters explain concepts and show configuration/code examples without requiring software installation.
- NITROS complexity: cuVSLAM NITROS internals are advanced. Chapter 2 must explain the performance benefit clearly without requiring students to understand CUDA memory management.

- **FR-013**: Each chapter MUST end with a Summary table of at least 7 key concepts.

### Key Entities

- **Isaac Sim**: NVIDIA's photorealistic robotics simulator built on Omniverse, using USD scene format, RTX ray tracing, and PhysX physics engine.
- **Synthetic Data**: Photorealistic, ground-truth-annotated training data generated entirely by a simulator — no real-world data collection required.
- **Replicator**: Isaac Sim's domain randomization tool that varies scene parameters (lighting, textures, poses, camera angles) to generate diverse synthetic datasets.
- **GEM (GPU-Enabled Module)**: A pre-built, hardware-accelerated Isaac ROS 2 package that processes sensor data on the GPU without CPU bottlenecks.
- **NITROS**: NVIDIA Isaac Transport for ROS — a zero-copy transport layer that passes tensor data between GPU-resident nodes without CPU-GPU memory copies.
- **cuVSLAM**: CUDA Visual SLAM — a GPU-accelerated Visual SLAM algorithm that estimates robot pose from camera images using feature extraction, matching, odometry, and loop closure.
- **Nav2**: The ROS 2 navigation stack; successor to ROS 1 move_base; provides costmap-based global path planning and local trajectory execution for autonomous robot navigation.

## Assumptions

- Students have completed Modules 1 and 2 before this module; ROS 2 topics and digital twin concepts are assumed knowledge.
- Software installation is out of scope — chapters are conceptual; no hands-on labs requiring Isaac Sim, Isaac ROS, or Nav2 installation.
- Isaac Sim 4.x (Omniverse-based) is the reference version; USD format and RTX rendering are key differentiators from Gazebo.
- Isaac ROS 2 (Humble-compatible) is the reference; GEM packages are the primary delivery mechanism.
- Nav2 with NavFn global planner and DWB (Dynamic Window Approach B) local planner/controller is the reference configuration.
- All content uses ASCII-only characters in code/config blocks (Windows cp1252 encoding safety).
- MDX v3 parsing rules apply: no bare XML angle brackets in prose or table cells.
- Relative links use `.md` extension in category index pages.

- Multi-robot coordination or fleet management
